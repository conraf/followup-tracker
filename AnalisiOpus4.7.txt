Analisi completa. Di seguito i suggerimenti organizzati per categoria, in ordine di priorità decrescente.

🔐 Sicurezza
✅ XSS parziale nei campi label (linee 887-890, 1019-1022): c.appLabel, c.f1label, c.f2label, c.f3label non passano da escHtml prima di finire in innerHTML. Oggi sono generati internamente ma un JSON importato maliziosamente può iniettare HTML arbitrario tramite questi campi.

Nessuna Content Security Policy: manca un <meta http-equiv="Content-Security-Policy">. Con una CSP stretta (script-src 'self' apis.google.com accounts.google.com) mitigheresti gran parte degli attacchi XSS.

Validazione assente sul Client ID salvato: salvaClientId accetta qualsiasi stringa. Aggiungere regex check ^\d+-[a-z0-9]+\.apps\.googleusercontent\.com$.

✅ Nessun limite di dimensione sull'import JSON: un file enorme può saturare localStorage (~5-10MB) e bloccare l'app. → Aggiunto controllo: file > 5MB viene rifiutato con alert.

Google Fonts caricati da CDN esterno: leggero leak di privacy (header Referer verso Google). Considerare self-hosting.

🐛 Funzionalità / Bug logici
✅ Statistiche "Followup Oggi" errate (linea 915): else if (d < oggi) oggiN++ conta anche gli overdue come "oggi". Dovrebbero essere categoria separata (o esclusi). → Aggiunta 4a card "Scaduti".

✅ Notifica "scadenza oggi" include i passati (linea 935): if (d <= oggi) mostra anche follow-up scaduti da giorni come "in scadenza oggi".

Token GCal scaduto non rilevato proattivamente: dopo 1h il token è invalidato lato Google ma _gcalToken resta valorizzato. Cliccare 📅 fallisce silenziosamente (solo il catch del 401 ripulisce). Andrebbe un check con scadenza stimata (expires_in restituito da Google) + timer.

✅ Duplicati calendario: se l'utente clicca 📅 due volte sulla stessa riga, crea eventi doppi. Nessun eventId custom o flag "inviato". → Aggiunto flag gcalSent + conferma al secondo click + stato ⏳ durante invio.

✅ Race condition ID: id: Date.now() su submit rapidissimi può collidere. Usare crypto.randomUUID().

_gcalToast condivide la notif-bar con le scadenze: se il toast si sovrappone a una notifica attiva, il testo viene salvato/ripristinato ma visivamente è confuso. Andrebbe un toast dedicato separato.

Submit validation di-tipo lasso: non c'è check che giornoApp sia nel futuro, non c'è check sull'ora in input.

🏗️ Qualità del codice
Monolite da 1270+ righe: separare in index.html, styles.css, app.js. Migliora cache del browser e manutenibilità.

Pattern ripetuto JSON.parse(localStorage.getItem(...)) || []: estrarre in helper (getChiamate(), setChiamate()) con try/catch per localStorage corrotto.

Nessun error handling su JSON.parse(localStorage...): se il dato è corrotto (utente manipola manualmente o bug passato), l'intera app crasha silenziosamente.

Magic strings: 'chiamate', 'chiamate_archivio', 'gcal_client_id' ripetuti ovunque. Usare const STORAGE_KEYS = {...}.

Inline onclick ovunque: mescola struttura e comportamento, rende il codice difficile da testare e limita l'applicabilità di una CSP stretta (servirebbe 'unsafe-inline'). Migrare a addEventListener.

Inline styles nell'header (linea 527-534), nel footer (692-696), nei button (637-639): sposterei tutto nel blocco <style>.

fmtData/fmtDT riscritte in due posti (762-769 e 803-804). DRY.

cancellatutte: naming inconsistente (tutto minuscolo vs camelCase del resto). Anche se richiede rename del listener HTML.

mostraDati ri-renderizza l'intera tabella ad ogni modifica. OK per liste piccole, ma con archivi grandi rallenta.

Nessun Service Worker: il manifest dichiara PWA ma senza SW non c'è cache offline né controllo versione. Aggiungendolo risolverebbe anche il problema dell'aggiornamento (Ctrl+Shift+R necessario oggi).

Nessun meccanismo di migrazione dati: se un giorno cambi schema di chiamata, i backup JSON vecchi potrebbero non importare. Versionare il JSON esportato ({ version: 1, data: {...} }).

Nessun test automatico.

♿ Accessibilità / UX
✅ Button emoji-only senza aria-label: 🗂️, 🗑️, 📅, 📊, 🗃️, 📂 sono invisibili per screen reader. Aggiungere aria-label="Archivia" ecc.

Mancanza role/aria-expanded su toggle (storico, guida rapida, pannello impostazioni GCal).

Conferma distruttiva con confirm() nativo: funziona ma esteticamente dissonante rispetto al resto dell'UI. Modale custom.

Colori status (arancione/verde/rosso) non hanno affidabilità per daltonici: affiancare sempre un'icona/testo.

✅ Nessuna indicazione visiva di caricamento durante invio a GCal: l'utente non sa se sta lavorando. → Bottone diventa ⏳ disabilitato durante l'invio.

Mobile: table ha min-width: 820px causando scroll orizzontale su telefono. Per uso mobile serio servirebbe layout a card.
